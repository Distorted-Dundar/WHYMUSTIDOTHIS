Objectives

  • Students will connect a local JavaScript file within their HTML.
  • Students will use document.querySelector to access DOM nodes in their JavaScript.
  • Students will use .addEventListener to add click events to their HTML elements.
  • Students will use .innerHTML to modify content on their page in response to user interactions.

Key vocabulary: DOM (Document Object Model), Selector, Event, Constant, Variable

INSTRUCTOR NOTE: Even though DOM (Document Object Model) is listed as key vocabulary, this could easily evolve into a very time-consuming and intimidating rabbit trail, and it’s not actually explicitly included anywehre in the lecture. To keep things moving, you’ll want to keep your exploration of the Document Object Model as lightweight as possible. Here’s one possible simplified explanation you can share with students:

The ‘DOM’ or Document Object Model is a way of thinking about a rendered HTML page - this lets you select individual pieces of your page and interact with them as JavaScript objects, even though they were originally written in HTML. At this stage in your journey, it’s not crucial that you understand every aspect of the DOM (though there is good documentation about the DOM if you want it); you just neeed to know that when we talk about the DOM, we’re usually talking about using JavaScript to interact with our HTML.

ADDITIONAL INSTRUCTOR NOTE: You’ll notice that this lesson uses arrow functions as callbacks for event listeners, but it does not make space for much conversation around any of those vocabulary words. You’re advised to touch briefly on the fact that the arrow function will be executed when the event occurs, but not to dive much deeper than that for now. Most of your students are likely to have studied functions in some capacity before this course, and so you’re encouraged to lean on that prior knowledge. If you do feel compelled to introduce functions more officially during this lesson try to emphasize just ONE feature of functions - talking about how they make our code more readable, more abstract, more DRY, and more modular all at once is generally overwhelming, and certainly not necessary to accomplish today’s objectives.

Demo

  1. Present the final version of the site, linked below.
    • make sure that you’re presenting your whole screen, otherwise the popups may not be visible to students.
  2. Click on the button labeled “Click to pet the baby goat”, and show students the pop-up.
  3. Click on the button labeled “Click to pet the bunny”, and show students that the text changes.
  4. Click on the button labeled “Click to feed the mama goat”, and show students that the carrot count goes up.
  5. Click on the button labeled “Pet the alligator anyways”, and show students that the content of the page changed.
  6. Ask students: “What is the big difference between this page and the page we built yesterday?”. Consider asking questions like “which animal’s button do you think has the most complicated code?”
    • Try to get the students to explain in their own words that this page is interactive.
    • Don’t spend more than a few minutes on this though.

Stepped Solution

  1. Fork the repl.it project which contains the lesson’s starter code into your own copy for use during lecture. You can do this by clicking on the project name in the upper-left of the repl.it UI, selecting the 3-dots menu, selecting “Fork”, and then clicking “Confirm”. This will create a new repl.it project with a copy of the starter code. This is the copy that you will edit during lecture.

  2. Share the starter code from your newly-remixed repl.it project with students, have them open the preview. They’re likely to notice that the page looks almost finished, but point out that right now none of the buttons do anything.

  3. Open the browser’s inspector and show students how to study the HTML. If you haven’t used the inspector at all in the first two days, you’ll want to take extra time to point out that if they mouse over an element in the inspector, the corresponding element will be shown in the browser. Demonstrate the “select an element” tool and show that you can use it to click on an element in the browser to see the corresponding HTML in the inspector.

  4. Name that there are roughly 50 elements on the page, and that none of those are added to JavaScript automatically. They will have to be explicitly selected if we want to bring them to life. Give students time to try to identify which elements we’ll need to select for interactivity. Students will likely identify the buttons, and will not likely realize we’ll also need the "status" paragraphs - that’s fine.

  5. Show students how to navigate to the JS console in their browsers. Make sure everyone is able to open their JS console before moving on, and pause briefly to explain that the console will be our primary JS debugging tool.

  6. Open the JavaScript file and point out that there should be a “script running” message logged to the console, but there isn’t just yet - that’s because we haven’t connected the JavaScript file to our HTML file. Find the script tag and add "script.js" to the src attribute so the JavaScript file runs. Confirm that everyone who is coding along sees that message before continuing.

  7. Point out that one element has already been selected (using a # to look by ID, just like in CSS) and stored in a constant. If you choose to discuss constants, do so only briefly. Then, add console.log(button1) and demonstrate how most browsers will not only log a lot of information about the selected element, but they will also highlight the corresponding rendered element on the page when you mouse over it in the console.

    • Using a # to select by ID will feel intuitive to students who have had HTML and CSS practice prior to this week, but if most of your students haven’t used CSS before, you may want to pause here and explain why that # is so important by contrasting it with something like querySelector(div).

  8. You can either repeat this process with querySelector() to select all four buttons, have students do this part independently for practice, or repeat this process each time you get ready to add another event. As long as you have emphasized the word “selected” each time it has been bolded in these instructions, students tend to take to querySelector() pretty intuitively.

  9. Add button1.addEventListener("click", e => {}). This is a LOT of code all at once, and it’s glossing over a fair bit. Students generally handle this perfectly well, but it can be helpful to have a planned think-aloud to work in key vocabulary as you type.

    • For example: “We’re going to take button1 and add an event listener to it. We’ll listen specifically for a click event, and that event will be fed into a function. We’re going to build that function out right here with an arrow, so that event will trigger this block of code between curly braces.”
    • NOTE: While this is technically a callback function, that vocabulary is generally not very helpful for students at this juncture. Consider glossing over that for now and coming back to it when that vocabulary feels more important.
      • If you still feel compelled to speak on functions, consider keeping your explanation very lightweight, e.g.: “If you’ve studied functions before, then the idea of what we’re doing here is probably familiar, but this arrow syntax may be new to you. Just know that function() {} and () => {} are, for our purposes today, essentially interchangeable - the arrow syntax is a bit newer. If you’re completely new to functions, you can ignore everything I just said. As long as you know that our click event is going to trigger the block of code after the arrow, you’ve got what you need to get through today’s lesson.”

  10. Inside the arrow function, add console.log("Button 1 pressed!") and show students that we have our first working button!

  11. Add alert("Meh-eh-eh-eh!") to that first button’s arrow function.

  12. Celebrate! The first button is done!

  13. Select button2 and store it in a new variable: const button2 = document.querySelector("#button2");

  14. Add button2.addEventListener("click", e => {}), add a console.log("Button 2 pressed!"), and then either ask or tell students what else (based on the finished example) should happen when button 2 is pressed.

  15. Since we’re going to display the results in the status2 paragraph, we’ll need to select that as well. Add const status2 = document.querySelector("#status2") to your script and then a console.log(status2) to confirm that it has been selected.

  16. We want to change the inside of the status2 paragraph, so we’ll do that using the .innerHTML property. This will leave the opening and closing tags intact, but reset the inside to whatever we specify. Add status2.innerHTML = "Thanks for petting the bunny! He looks really happy!" to your arrow function for button 2.

  17. Ask or tell students what’s different about the third button based on what they saw in the finished version. Name that we’re going to need a counter to keep track of the number of carrots. Briefly discuss let as a way to initialize variables, and add let carrots = 2 to start off with the correct number.

  18. Capture status3 in a constant with a .querySelector(), and console.log it out to make sure it worked.

  19. Add the event listener for button 3, test that it’s working with a console.log("Button 3 pressed!").

  20. Add carrots = carrots + 1 and describe what that will do, and then immediately add status3.innerHTML = "Coco has had " + carrots + " carrots today!" and discuss what will happen here. Since this is our first exploration of concatenation in this course, it might be worth discussing Strings and how that + sign is assumed to mean “glue these two things together” whenever JavaScript can’t reasonably sum them.

  21. Ask or tell students what’s different about the fourth and final button.

  • Note that the fourth button is really just additional practice. If there isn’t time, the fourth button could reasonably be something that some students get to as part of independent practice.

  1. Since we’re going to be manipulating a much larger chunk of the page, have students add const zoo = document.querySelector(".container"). Then add console.log(zoo) to confirm that we’ve done it.

  2. Add the fourth button’s event listener and a console.log statement inside it to confirm that the event is firing.

  3. Add the below line of code to the event listener’s arrow function.

1


										
  zoo.innerHTML = `<h1 class="title">Due to some dangerous rulebreaking, the petting zoo is temporarily closed</h1>`;

  1. It’s worth pointing out that we’re using backticks for this string instead of quotation marks like we used for the rest. Point out that you can use single quotes, double quotes, and backticks to denote a String in JavaScript, and award a big high five to any student who can figure out why we can’t use double quotes here.

  2. Make sure to share the repl.it project that you’ve been working in during the lecture with the students. That will make it easier for anyone who fell behind to get to the same state you’re at. You can do this by just copying & pasting the URL from your browser window.

Extensions for Independent Practice

Mild

  • Add a fifth animal! Model it after whichever of the four example animals whose behavior you want to replicate.
  • Change at least one of the example animals to an animal of your choice, like a hamster or a lion. Modify the HTML to reflect that, and modify the event listener as well.

Medium

  • Right now the mama goat can eat infinite carrots. That’s not too realistic. Find a way to make her respond differently if she gets too many carrots. Here are some ideas of what that could look like:
    • Change the message to be something about the dangers of overfeeding, or to show that she doesn’t accept carrots once she’s full.
    • Disable or remove the button.
    • Create an alert.

Spicy

  • Add another button to each of the animals, and code out corresponding behaviors. The goal is to approximate a pet-keeping simulation. Here are some ideas for what those buttons might look like:
    • “Play”, which increases a happiness variable, but decreases an energy variable.
    • “Feed”, which increases energy and fullness.
    • “Brush”, which increases happiness and cleanliness.
  • Break the alligator button into multiple steps - when you first click it, change the button to say “are you sure?” or something similar.
